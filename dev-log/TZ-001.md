## Контекст
В multi-way SUM-режиме добавлена функция автоматического выравнивания задержек
между полосами. Алгоритм: последовательный sweep LF→HF, минимизация weighted MSE
когерентной суммы vs Σ target в зоне ±1 октава от кроссовера.
Код уже написан (auto-align.ts + изменения в FrequencyPlot.tsx), но не проверен
в рантайме и не сопровождён инкрементом билда.

## Пост-фиксы

### Fix 1 (промт 2)
- **Проблема:** Задержки отрицательные (−0.36, −0.39 ms), sweep ±2 ms мал для НЧ кроссоверов, сортировка нестабильна
- **Причина:** Алгоритм не нормализовал задержки к положительным, фиксированный sweep range
- **Решение:** Нормализация min→0, адаптивный sweep (5ms для <200Гц, 3ms для <500Гц), стабильная сортировка по bandIndex
- **Промт:** фикс auto-align.ts — три конкретных изменения

### Проблема 2 (не промт — пользователь нашёл)
- **Проблема:** AUTO на неоптимизированных полосах даёт бессмысленные задержки и провалы в сумме
- **Причина:** Corrected-кривые без PEQ = сырые замеры + XO фильтры → задержки оптимизируются под некорректные данные
- **Решение:** В процессе — нужна защита: AUTO только после PEQ-оптимизации всех полос

### Побочный ущерб
- **Проблема:** Чекбоксы CORR+XO пропали у части полос в SUM-матрице
- **Причина:** Вероятно связано с изменениями в FrequencyPlot.tsx (dlSuffix в label или autoAlignCache блок)
- **Решение:** Ожидает диагностики

## Lessons Learned
- **Паттерн ошибок:** Алгоритм оптимизации работает на данных которые могут быть невалидными — нужны preconditions
- **Правило в шаблон ТЗ:** Всегда описывать preconditions для новых фич (какие данные должны быть готовы до запуска)
- **Правило в шаблон ТЗ:** Промт должен требовать проверку побочных эффектов на UI-элементах рядом с изменённым кодом
