# Dev Tracker — трекер ТЗ и качества разработки

Этот скил управляет полным циклом отслеживания разработки в проекте PhaseForge.
Все данные хранятся в `dev-log/` в корне проекта (рядом с `src/`, `src-tauri/`).

---

## Когда активируется

- Пользователь хочет **создать ТЗ** (промт для сессии в Code)
- Пользователь вернулся из Code и хочет **зафиксировать результат**
- Пользователь хочет **посмотреть историю** ТЗ или тренды качества
- Пользователь спрашивает о прошлых ошибках или паттернах

---

## Структура хранения

```
dev-log/
├── index.md          — сводная таблица всех ТЗ
├── TZ-001.md         — первое ТЗ
├── TZ-002.md         — второе ТЗ
└── ...
```

---

## Режим 1: Создание ТЗ

Когда пользователь обсудил задачу и готов сформировать промт для Code.

### Шаги

1. **Определить ID** — прочитать `dev-log/index.md`, найти последний ID, взять +1
2. **Прочитать историю** — загрузить секции "Lessons Learned" из последних 3–5 TZ-файлов
   (или всех, если их меньше). Выделить повторяющиеся паттерны ошибок и правила.
   Учесть эти правила при составлении промта — каждый промт должен
   быть "умнее" предыдущего за счёт накопленного опыта.
3. **Создать файл** `dev-log/TZ-{NNN}.md` по шаблону ниже
4. **Обновить** `dev-log/index.md` — добавить строку в таблицу
5. **Показать пользователю** готовый промт для копирования в Code
   (в начале промта кратко перечислить учтённые lessons learned,
   чтобы пользователь видел что опыт применён)

### Шаблон TZ-файла (создание)

```markdown
---
id: TZ-{NNN}
title: {краткое название фичи}
build: {текущий билд → +1}
date: {YYYY-MM-DD}
status: pending
---

## Контекст
{зачем это нужно, DSP/бизнес-логика}

## Задача
{конкретные файлы, функции, логика изменений}

## Номер сборки
Инкремент текущего билда. Определить текущий номер в tauri.conf.json,
увеличить на 1, обновить: tauri.conf.json (version + title), lib.rs (startup log).

## Критерий готовности
{как проверить что работает — конкретные шаги}

## Проверка регрессий
{что не должно сломаться — какие табы, фичи, вьюхи проверить}

## Сборка и проверка
1. `cargo tauri dev` — компилируется без ошибок
2. {конкретные шаги проверки фичи в UI}
3. {проверка других табов/вьюх}
4. При успехе: `git add`, `git commit -m "feat: {title} (b{NN})"`, `git push`

---
## Промт для Code

{полный текст промта, готовый к вставке в сессию Code}

---
## Результат
_заполняется после исполнения_

## Пост-фиксы
_заполняется после исполнения_

## Lessons Learned
_заполняется после исполнения_
```

### Шаблон index.md

Если файл не существует, создать:

```markdown
# PhaseForge — Dev Log

| ID | Дата | Билд | Название | Статус | Промт | Исполн. | Допромты | Ущерб |
|----|------|------|----------|--------|-------|---------|----------|-------|
```

Новая строка при создании ТЗ:

```
| TZ-001 | 2026-04-12 | b102 | Auto-align delays | pending | — | — | — | — |
```

## Режим 2: Фиксация результата

Когда пользователь вернулся из Code и даёт фидбек.

### Что спросить у пользователя

1. **Сработало с первого промта?** (да/нет)
2. **Оценка качества промта** (1-5): насколько точно ТЗ описало задачу
3. **Оценка исполнения** (1-5): как Code справился
4. **Сколько допромтов** потребовалось до результата
5. **Побочный ущерб**: что сломалось (если ничего — записать "нет")
6. **Пост-фиксы**: для каждого допромта — проблема, причина, решение, текст промта
7. **Lessons learned**: обобщение — паттерн ошибок, что добавить в шаблон ТЗ

### Шаги

1. Открыть соответствующий `dev-log/TZ-{NNN}.md`
2. Заполнить секции "Результат", "Пост-фиксы", "Lessons Learned"
3. Обновить `status` в frontmatter: `done` / `failed` / `partial`
4. Обновить строку в `dev-log/index.md` — заполнить оценки

### Формат секции "Результат"

```markdown
## Результат
- Качество промта: {1-5}/5
- Исполнение: {1-5}/5
- Допромтов: {N}
- Побочный ущерб: {описание или "нет"}
```

### Формат секции "Пост-фиксы"

```markdown
## Пост-фиксы

### Fix 1
- **Проблема:** {что сломалось / не работает}
- **Причина:** {почему — промт неточный, не учтён контекст, регрессия, баг Code}
- **Решение:** {что допромтили / исправили}
- **Промт:** `{текст допромта если важен}`

### Fix 2
...
```

### Формат секции "Lessons Learned"

```markdown
## Lessons Learned
- **Паттерн ошибок:** {обобщение — что пошло не так и почему}
- **Правило в шаблон ТЗ:** {конкретное дополнение к формату будущих ТЗ}
```

### Обновление index.md

Заполнить пустые ячейки в строке:

```
| TZ-001 | 2026-04-12 | b102 | Auto-align delays | done | 4/5 | 3/5 | 2 | сломал WAV |
```

## Режим 3: Обзор истории

Когда пользователь хочет посмотреть тренды или вспомнить прошлый опыт.

### Что показывать

1. **Таблица** из `index.md` — все ТЗ, оценки, статусы
2. **Средние оценки** — промт и исполнение, тренд по последним 5-10
3. **Частые паттерны ошибок** — собрать все Lessons Learned, выделить повторяющиеся
4. **Рекомендации** — на основе паттернов предложить изменения в шаблоне ТЗ

---

## Режим 4: Ретроспектива (каждые 5–10 ТЗ или по запросу)

Периодическое обобщение накопленного опыта — стратегический уровень обучения.

### Когда запускать

- Автоматически предложить после каждого 5-го зафиксированного ТЗ (TZ-005, TZ-010, ...)
- По явному запросу пользователя ("ретроспектива", "что мы выучили")

### Шаги

1. **Прочитать все TZ-файлы** — собрать все секции Lessons Learned и Пост-фиксы
2. **Выявить устойчивые паттерны:**
   - Какие типы задач дают больше всего фиксов?
   - Какие причины повторяются (неточный промт, регрессия, забыли проверить X)?
   - Есть ли прогресс — средняя оценка растёт или нет?
3. **Сформировать новые правила** для шаблона ТЗ
4. **Обновить MEMORY.md** в auto-memory — записать обобщённые правила,
   чтобы они были в контексте каждой будущей сессии Cowork.
   Формат: файл `feedback_tz_rules.md` с типом `feedback`,
   содержащий конкретные правила вида
   "всегда указывать X в промте потому что Y"
5. **Показать пользователю** сводку: тренды, паттерны, новые правила

### Что попадает в MEMORY.md (долгосрочная память)

Только обобщённые правила, а не конкретные ТЗ. Примеры:

- "Промт обязательно должен указывать какие табы проверить на регрессию — в 3 из 5 случаев регрессия была на непроверенном табе"
- "Для UI-фичей всегда включать в промт: стиль CSS класса, размер, позицию"
- "FIR-изменения всегда требуют проверки WAV-экспорта"

Это гарантирует что знания не теряются между сессиями.

---

## Важные правила

- **Не кодить.** Этот скил только создаёт/обновляет markdown-файлы в `dev-log/`.
  Код пишется исключительно через промты в Code.
- **Не угадывать оценки.** Все числовые оценки — только от пользователя.
- **ID строго инкрементален.** Читать index.md для определения следующего номера.
- **Промт для Code — полностью автономный.** Он должен содержать весь необходимый контекст,
  чтобы Code мог выполнить задачу без дополнительных вопросов.
- **Сохранять фактический текст допромтов** в пост-фиксах — это данные для обучения.
